# Chapter 13: Offline Sync & Caching

## 13.1 Offline-First Philosophy

The FOCUS platform is designed as an offline-first application. This is not a secondary feature or an afterthought — it is the foundational architectural principle that governs every decision about data storage, network communication, and user experience. The rationale is straightforward: cognitive training sessions are time-sensitive, deeply personal, and must never be interrupted by network conditions. A user who has carved out fifteen minutes for focus training must be able to complete that session regardless of whether they are on a plane, in a basement gym, in a rural area with intermittent connectivity, or simply experiencing a temporary network outage.

### 13.1.1 Core Invariants

The following invariants must hold at all times under all network conditions:

1. **Complete functionality offline.** Every game, every calibration routine, every scoring algorithm, every progression system, every achievement check, and every UI screen must function identically whether the device is online or offline. There is no degraded mode. There is no "limited functionality" state. The application is fully operational at all times.

2. **No network errors for core operations.** The user must never see a toast, dialog, banner, or any other UI element that says "no internet connection" or "network error" when performing core platform operations. If a network operation fails, it must fail silently in the background and retry when connectivity returns. The only acceptable network errors are for operations that genuinely require real-time server coordination (e.g., sending a live challenge to a friend who is currently online).

3. **Data never lost.** Every piece of data generated by the user — every session result, every score, every calibration, every settings change — must persist locally before any network operation is attempted. If the application crashes, the device loses power, or the network fails mid-operation, all locally-generated data must survive and be synchronizable when conditions permit.

4. **Deterministic conflict resolution.** When the same data is modified on two different devices (or on one device while offline and another while online), the conflict resolution must be deterministic. The same conflict must always resolve the same way, regardless of which device initiates the sync. There must be no race conditions, no flip-flopping values, and no data corruption from concurrent modifications.

5. **Seamless transitions.** The user must not be able to tell whether they are online or offline based on the application's behavior. The UI, the data, the responsiveness — all must be identical. The only difference is that certain background operations (sync, social features, leaderboards) may be slightly stale when offline.

### 13.1.2 Why Offline-First

The decision to go offline-first is driven by several concrete technical and business considerations:

- **User trust.** If a user completes a personal best in a focus game and the app fails to save it because of a network glitch, that user will never trust the app again. Data persistence is the foundation of user trust.

- **Session integrity.** Cognitive training sessions are structured experiences with precise timing requirements. Any network-induced latency or interruption during a session would corrupt the timing data and invalidate the results.

- **Global audience.** FOCUS serves users worldwide, including regions with unreliable internet infrastructure. An offline-first architecture ensures equitable access to the platform's core value proposition.

- **Mobile usage patterns.** A significant portion of usage occurs on mobile devices in transit — on buses, trains, subways, and planes. These environments have intermittent or no connectivity.

- **Battery life.** Constant network polling drains battery. An offline-first approach allows the application to minimize network operations and batch them intelligently.

### 13.1.3 What Requires Network

The following operations are the exception — they genuinely require network connectivity:

- Authentication (initial login, token refresh)
- Social features (sending challenges, viewing friend activity, real-time leaderboards)
- Content downloads (new games, updates, assets)
- Account management (profile changes that must propagate to all devices)
- Payment processing
- Analytics transmission (background, non-blocking)

When any of these operations is attempted without connectivity, the operation is queued and executed when connectivity returns. The user is informed through subtle UI indicators (described in Section 13.7) but is never blocked from using the application.

## 13.2 Storage Architecture

The FOCUS platform employs a layered storage architecture, with each layer optimized for a specific type of data and access pattern. The layers are not redundant — they each serve a distinct purpose, and data flows between them according to well-defined rules.

### 13.2.1 IndexedDB via Dexie.js (Primary Local Database)

**IndexedDB** is the primary local storage mechanism for the FOCUS platform. IndexedDB is a browser API that provides a transactional, asynchronous key-value store with support for structured data, indexes, and large storage quotas (typically 50% of available disk space, with a minimum of 500MB on most browsers).

**Why IndexedDB over alternatives:**

- **localStorage** is synchronous, blocks the main thread, and has a 5-10MB quota. It is unsuitable for any data beyond trivial preferences.
- **WebSQL** is deprecated and was never standardized.
- **Cache API** is designed for network response caching, not structured data storage.
- **OPFS (Origin Private File System)** is a newer API that provides file-system-like access. It is excellent for large binary data but lacks the query capabilities needed for structured data. It will be used in the future for game asset storage.

**Why Dexie.js over raw IndexedDB:**

Raw IndexedDB has a notoriously difficult API. It uses event-driven callbacks, requires explicit transaction management, and has no query language. Dexie.js provides:

1. **Type-safe schema definitions.** Dexie allows TypeScript interfaces to be directly mapped to database tables, providing compile-time type checking for all database operations.

2. **Simplified API.** Instead of the raw IndexedDB ceremony of opening a transaction, getting an object store, creating a request, and handling success/error callbacks, Dexie provides a clean chainable API: `db.sessions.where('status').equals('pending').toArray()`.

3. **Index support.** Dexie makes it trivial to define compound indexes, multi-entry indexes, and unique constraints — critical for efficient queries on sync status, timestamps, and game identifiers.

4. **Upgrade management.** Dexie handles database schema migrations automatically, with a clean upgrade callback mechanism that allows data transformation during version upgrades.

5. **Performance.** Dexie is optimized for bulk operations, with `bulkPut`, `bulkAdd`, and `bulkDelete` methods that significantly outperform individual operations.

6. **Where clause optimization.** Dexie's where clauses are optimized to use indexes, avoiding full table scans on large datasets.

### 13.2.2 SQLite via sql.js (Complex Queries)

For operations that require complex SQL queries — statistical calculations, ranking algorithms, full-text search — the platform uses sql.js, a JavaScript implementation of SQLite compiled to WebAssembly.

**Use cases:**

- **Statistics calculations.** Computing running averages, standard deviations, percentile rankings, and trend analyses across hundreds or thousands of session records. These queries involve GROUP BY, window functions, and complex WHERE clauses that are awkward to express in IndexedDB's query API.

- **Leaderboard ranking.** Computing a user's rank across all users for a specific game requires a full table scan with ORDER BY and LIMIT. While this can be done in IndexedDB, SQLite handles it more efficiently for large datasets.

- **Search.** Full-text search across session history, achievements, and game data requires FTS (Full-Text Search) capabilities that IndexedDB does not provide natively.

- **Aggregation.** Complex aggregation queries (e.g., "average score per day for the last 30 days, grouped by game") are natural in SQL and cumbersome in IndexedDB.

**Implementation approach:**

sql.js operates on a copy of the data extracted from IndexedDB. The data is loaded into an in-memory SQLite database, queried, and the results are returned. This is not a primary data store — it is a query engine that operates on a snapshot of the IndexedDB data. The data flow is:

1. IndexedDB is the source of truth.
2. When complex queries are needed, relevant data is extracted from IndexedDB.
3. Data is loaded into sql.js in-memory database.
4. SQL queries are executed.
5. Results are returned to the application.
6. The sql.js instance is destroyed (freeing memory).

The sql.js instance is never kept alive longer than necessary. It is created on demand, used, and then garbage collected. This prevents memory leaks and ensures that the in-memory database does not consume resources when not actively being queried.

### 13.2.3 localStorage (Preferences Only)

localStorage is used exclusively for storing user preferences and UI state that must survive page reloads but does not require complex queries or large storage. Examples include:

- Theme preference (light/dark/system)
- Language preference
- Last viewed screen
- Onboarding completion status
- Feature flag overrides
- Consent preferences (analytics, cookies)
- Last sync timestamp

localStorage is explicitly NOT used for:
- Game session data (too large, too important)
- Scores or records (must be transactional)
- User profile data (must be synced)
- Any data that exceeds 1KB in size

The restriction to preferences-only is enforced by code review conventions and linting rules. A custom ESLint rule prevents direct `localStorage.setItem` calls outside of the designated settings module.

### 13.2.4 Cache API (Network Response Caching)

The Cache API is used by the Service Worker to cache network responses — API responses, images, fonts, and other static assets. The Cache API is separate from IndexedDB and is managed entirely by the Service Worker lifecycle.

The Cache API provides:
- Named caches with versioning
- Request/response pair storage
- Batch operations (put, match, delete, keys)
- Integration with Service Worker fetch events

Cache storage is partitioned by type:
- `static-assets-v{hash}` — Precached app shell (HTML, CSS, JS)
- `runtime-api-v{hash}` — API response cache
- `runtime-images-v{hash}` — Image cache
- `runtime-fonts-v{hash}` — Font cache
- `game-assets-v{hash}` — Game-specific assets

### 13.2.5 OPFS (Origin Private File System) — Future

OPFS is a newer browser API that provides a private file system for each origin. It is being evaluated for future use in storing large binary data such as:

- Game asset bundles (textures, audio files)
- SQLite database files (for persistent sql.js usage)
- Export files (user data exports, reports)
- Offline-first analytics data (large volume)

OPFS is not yet used in the current architecture because browser support is still limited (no Safari support as of 2024), the sync access handle API requires a dedicated worker, and the existing IndexedDB plus Cache API combination meets current needs. OPFS will be adopted when Safari adds support and when the platform needs to store binary data larger than what IndexedDB handles efficiently.

## 13.4 Sync Engine

The sync engine is the core component responsible for maintaining consistency between the local IndexedDB database and the remote Supabase database. It implements a bidirectional sync protocol with conflict detection, resolution, and a priority-based queue system.

### 13.4.1 Sync Triggers

The sync engine responds to multiple triggers, each designed to balance data freshness with resource consumption:

1. **Automatic sync on connection restore.** When the browser's `navigator.onLine` status changes from `false` to `true`, or when a network request succeeds after a period of failures, the sync engine initiates a full sync cycle. This is the primary mechanism for ensuring that offline-generated data reaches the server.

2. **Periodic sync when online.** While the application is online and in the foreground, the sync engine runs on a configurable interval (default: 30 seconds). This interval is adjusted based on network quality (see Section 13.4.5). The periodic sync ensures that changes made on other devices are pulled down regularly.

3. **Manual sync (pull-to-refresh).** The user can trigger a manual sync by performing a pull-to-refresh gesture on applicable screens. This forces an immediate full sync cycle and is useful when the user wants to see the latest data from other devices.

4. **Foreground sync.** When the application transitions from background to foreground (detected via the `visibilitychange` event), the sync engine checks if a sync is due and initiates one if necessary. This ensures that data is fresh when the user returns to the application.

5. **Post-mutation sync.** After any local data mutation (completing a session, earning an achievement, changing settings), the sync engine adds the mutated entity to the sync queue and initiates a sync if online. This provides the lowest latency for pushing local changes to the server.

### 13.4.2 Sync Protocol

The sync protocol follows a structured four-phase sequence:

**Phase 1 — Preparation:**

The sync engine acquires a lock to prevent concurrent sync cycles, captures the local timestamp, retrieves pending items from the `local_sync_queue` ordered by priority, and separates items into push items (local changes to send) and the pull-since timestamp (last successful sync timestamp).

**Phase 2 — Pull (Download Server Changes):**

The engine sends a GET request to `/api/sync?since={last_sync_timestamp}`. The server returns all records modified since the timestamp. For each server record, the engine checks if the record exists locally. If it exists and the `server_version` matches, the local record is overwritten (server is newer). If the versions differ, a conflict is detected and the resolution strategy is applied. If the record does not exist locally, it is inserted as a new record. The `last_sync_timestamp` is updated locally.

**Phase 3 — Push (Upload Local Changes):**

The engine sends a POST request to `/api/sync` with the pending items and the client timestamp. The server processes each item: for creates, it inserts if the record does not exist and returns a new `server_version`; for updates, it checks the version and updates if matching or returns a conflict if not; for deletes, it performs a soft delete and returns confirmation. The server returns a results array with each item's status. For successes, the local record is updated with the server version and marked as synced. For conflicts, an entry is created in `local_conflicts` and the resolution strategy is applied. For errors, the item is marked as failed and scheduled for retry with exponential backoff.

**Phase 4 — Cleanup:**

Completed items are removed from the `local_sync_queue`, the `last_sync_timestamp` is updated to the server timestamp, the sync lock is released, and the UI is notified of the sync completion status.

### 13.4.3 Conflict Resolution Strategy

Conflict resolution varies by entity type, reflecting the different semantics of each data type:

**Game Sessions — Server Wins.** Game sessions are append-only records. Once created and completed, they should never be modified. Conflicts are nearly impossible because sessions use UUID v4 identifiers and are never modified after completion. If a conflict occurs, the server version is accepted and logged for investigation.

**User Settings — Last-Write-Wins (Server Timestamp).** User settings are independent key-value pairs. The conflict resolution is last-write-wins based on the server timestamp. This is a deliberate simplification: settings are small, independent, and users rarely change the same setting on multiple devices simultaneously.

**Achievements — Server Wins.** Achievements are validated server-side. The server checks achievement conditions, verifies underlying data, and issues the achievement. The server version always wins to prevent achievement duplication, false unlocks, and inconsistent state.

**Scores — Server Wins.** Scores must be server-validated to prevent cheating. The server recalculates scores from raw event data and rejects any score that does not match. In conflicts, the server version is authoritative.

**Profile Data — Field-Level Merge.** User profile data (display name, avatar URL, bio) uses field-level last-write-wins. Each field has its own implicit timestamp, and the latest timestamp wins per field. This requires field-level comparison rather than whole-record comparison.

**Friend Requests — Server Wins.** Friend requests are managed by the server's social graph logic. Conflicts are resolved by the server version to maintain consistency.

### 13.4.4 Conflict Detection

Conflict detection relies on version numbers. Each record has a `server_version` field that is incremented on every update. The algorithm compares `local_record.server_version` with `server_record.server_version`: if they match, there is no conflict and the server record overwrites the local one; if the local record was previously synced (`sync_status === 'synced'`), the server changed since the last sync and the server wins; if the local record is pending sync (`sync_status === 'pending'`), both sides changed and a conflict is detected.

When a conflict is detected, the system logs: entity type, entity ID, local data, server data, local version, server version, local modified timestamp, server modified timestamp, the resolution applied, the resolution timestamp, and whether the resolution was synced.

### 13.4.5 Sync Priority Queue

The sync queue is ordered by priority to ensure the most important data synchronizes first:

| Priority | Entity Type | Rationale |
|----------|-------------|-----------|
| 1 | Game sessions | Most important — user just earned them, core value |
| 2 | Scores and records | High value — leaderboard positioning, personal bests |
| 3 | Achievements | Medium-high value — motivational, less time-sensitive |
| 4 | User settings | Medium value — preferences, not critical |
| 5 | Analytics events | Low value — important for product but not user-facing |
| 6 | Social actions | Lowest priority — friend requests, challenge invites |

Within each priority level, items are ordered by creation time (FIFO). A priority-1 item created an hour ago syncs before a priority-5 item created a minute ago.

### 13.4.6 Bandwidth Awareness

The sync engine adapts its behavior based on the user's network conditions using the Network Information API (`navigator.connection.effectiveType`):

| Network Type | Sync Interval | Batch Size | Description |
|-------------|---------------|------------|-------------|
| slow-2g | 300 seconds | 10 items | Extremely limited bandwidth |
| 2g | 300 seconds | 20 items | Limited bandwidth |
| 3g | 60 seconds | 50 items | Moderate bandwidth |
| 4g | 30 seconds | 100 items | Good bandwidth |
| wifi | 30 seconds | 200 items | Excellent bandwidth |

When the Network Information API is unavailable (Firefox, Safari), the engine defaults to 30-second intervals with 100-item batches.

**User overrides:** Always sync (default, enabled), WiFi only (for limited data plans), or Manual only (user triggers sync explicitly).

### 13.4.7 Retry and Backoff

Failed sync operations use exponential backoff: 5 seconds, 15 seconds, 45 seconds, 2 minutes, 5 minutes, then capped at 10 minutes. After 10 failed attempts, the item is marked as `failed` and the user is notified through the sync status indicator. The user can manually retry or ignore the failed item.

## 13.5 Caching Strategy

### 13.5.1 Static Assets (HTML, CSS, JS)

**Strategy: Cache First, Network Fallback (Service Worker).** The Service Worker intercepts all requests for static assets and serves them from cache when available. The cache is populated during the `install` event (precache) and updated during `activate`.

Cache naming uses `static-assets-v{content-hash}`. The content hash changes whenever any static asset changes, ensuring old caches are automatically invalidated. The precache manifest includes `index.html`, `main.{hash}.js`, `vendor.{hash}.js`, `styles.{hash}.css`, critical fonts, and app icons. Non-critical assets are cached lazily on first use.

The update mechanism works as follows: the new Service Worker is installed in the background and enters a `waiting` state. When all tabs using the old Service Worker close, the new one activates, old caches are cleaned, and the user can optionally click an "Update available" toast to immediately activate.

### 13.5.2 API Responses

**Strategy: Stale-While-Revalidate (React Query).** React Query v5 manages API response caching with a default `staleTime` of 5 minutes and `gcTime` (garbage collection) of 30 minutes. Queries refetch on window focus and network reconnect. Retry is configured with 3 attempts and exponential backoff.

Query-specific stale times: game data (5 minutes), leaderboards (1 minute), user profile (5 minutes), friend list (2 minutes), achievements (10 minutes), real-time data (0 seconds — always refetch).

**Optimistic updates** apply mutations to the local cache immediately before server confirmation. If the server rejects the mutation, the update is rolled back. This provides instant UI feedback while maintaining data integrity.

### 13.5.3 Images

**Strategy: Cache First with Network Fallback, LRU Eviction.** Images are cached by the Service Worker with a maximum cache size of 100MB. When exceeded, the oldest 20% of responses are evicted, bringing the cache below 80%. Eviction runs during idle periods via `requestIdleCallback`. All images outside the viewport use lazy loading with Intersection Observer and a 200px root margin for preloading.

### 13.5.4 Game Assets

**Strategy: Precache on First Load, IndexedDB Storage.** Game assets are stored as binary blobs in IndexedDB. On first game load, all assets are downloaded and stored. Subsequent loads read from IndexedDB. On app update, a manifest check identifies changed assets for incremental re-download. Total game asset storage is capped at 500MB, with LRU eviction for games not played in 30+ days.

### 13.5.5 Database Query Caching

Statistics are cached in `local_statistics` with a 1-hour TTL, recomputed lazily on next access after expiration. Leaderboards are cached in `local_leaderboards` with a 5-minute TTL, refetched from the server when online or served stale when offline. Search queries execute against the local sql.js database on demand without caching.

## 13.6 Service Worker

### 13.6.1 Workbox Configuration

The platform uses Workbox for Service Worker management. The configuration precaches the app shell, uses Stale-While-Revalidate for API responses, CacheFirst for images and fonts, and BackgroundSyncPlugin for failed sync requests. Expiration plugins enforce maximum entry counts and age limits for each cache.

### 13.6.2 Lifecycle

**Install:** Precache the app shell from the build-time manifest. `skipWaiting()` ensures immediate activation.

**Activate:** Clean old caches (any cache not matching the current version hash). Claim all open clients so the new Service Worker controls them immediately.

**Fetch:** Intercept all fetch events. Check if the request matches a registered route. Apply the route's caching strategy or pass through to the network. If the network fails, return the cached response or a fallback.

**Message:** Communicate with the main thread via `postMessage` for sync coordination and update notifications.

### 13.6.3 Update Mechanism

On app foreground, the application checks for Service Worker updates with `updateViaCache: 'none'`. If a new Service Worker is detected, a subtle toast notification appears. The user clicks to reload all tabs and activate the new Worker. If dismissed, the update takes effect when all old tabs close. For critical security patches, the update can be forced by sending a `SKIP_WAITING` message.

## 13.7 Offline UI

### 13.7.1 Online/Offline Indicator

A small 6px dot in the application header indicates connectivity: green (online and synced), yellow (syncing), gray (offline), red (sync errors). The dot pulses subtly during sync. It is intentionally small and non-intrusive.

### 13.7.2 Sync Status Indicator

On screens displaying server-dependent data, a footer indicator shows: "Syncing..." with a spinner, "Synced just now", "Synced 5 minutes ago" (stale), or "3 items pending". Hidden on purely local screens.

### 13.7.3 Offline Mode Banner

When offline, a banner states: "You are offline. Your progress is saved locally and will sync when you reconnect." Auto-dismisses after 5 seconds. Reappears only on state transition (online to offline), not on every screen change.

### 13.7.4 Pending Items Badge

A small badge on the sync status indicator shows the count of pending items. Hidden when count is 0. Red when count exceeds 20 (potential sync issue).

### 13.7.5 Conflict Resolution Dialog

When a conflict requires user intervention (rare — profile data conflicts where field-level merge is not possible), a dialog presents options: keep this device's version, keep server version, or show differences. All other entity types resolve automatically.

## 13.8 Data Integrity

### 13.8.1 Checksums

Every record includes a SHA-256 checksum computed from its sorted content. The checksum is recalculated on every write and verified on every read. Mismatches trigger recovery from backup or server.

### 13.8.2 Transaction-Based Writes

Critical multi-table operations (session completion, achievement unlock) use Dexie transactions to ensure all-or-nothing semantics. If any operation within the transaction fails, all operations are rolled back.

### 13.8.3 Backup and Restore

The local database is exported to compressed JSON periodically (weekly, when idle), on manual trigger, and pre-migration. Restore is triggered on corruption detection, schema migration failure, or user request.

### 13.8.4 Schema Migration

Migrations use Dexie's versioning system. They are forward-only, idempotent, preserve all existing data, tested against production data copies, and automatically preceded by a backup.

## 13.9 Performance Considerations

### 13.9.1 IndexedDB Performance

Bulk operations (`bulkPut`, `bulkAdd`) are 10-100x faster than individual operations. Transaction scope is minimized to avoid blocking. Queries always use indexed fields. Cursors are used for large result sets to avoid loading all records into memory.

### 13.9.2 Sync Performance

The sync engine runs in a Web Worker to avoid blocking the UI thread. Only changed records are synced (incremental). Sync payloads are compressed with gzip. Consecutive sync triggers are coalesced into single cycles.

### 13.9.3 Cache Performance

Critical assets are precached during installation. Non-critical assets are cached on demand. Cache size limits prevent unbounded growth. Cache partitioning by type prevents eviction of frequently-accessed data by rarely-accessed data.

## 13.10 Testing the Offline/Sync System

### 13.10.1 Offline Testing

Offline behavior is tested via Chrome DevTools network throttling, Playwright `page.route` for simulating network failures, and custom Service Worker interception utilities.

### 13.10.2 Sync Testing

Sync is tested with a mock server simulating conflicts, failures, and slow responses. Time manipulation fast-forwards periodic sync and TTL expiration. Multiple browser contexts simulate multi-device scenarios.

### 13.10.3 Conflict Testing

Explicit test scenarios include: same record modified on two devices, concurrent creation with identical logical keys, delete-modify conflicts, and network partition with rejoin.

## 13.11 Summary

The offline sync and caching architecture rests on three pillars: IndexedDB as the source of truth ensuring data is never lost; bidirectional sync with deterministic conflict resolution maintaining consistency across devices; and multi-layered caching from Service Workers to React Query to IndexedDB optimizing every access pattern. This architecture ensures FOCUS provides a seamless, reliable experience regardless of network conditions.
